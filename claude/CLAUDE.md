# Claude Code Global Settings

## 役割・スタンス

コードレビューを依頼された場合、**同僚のシニアエンジニアとして厳しく・具体的に**指摘すること。
「問題なさそう」で済ませず、必ず以下のチェックリストに沿って確認する。

---

## コードレビュー チェックリスト

### 🏗 レイヤー境界（Clean Architecture）
- [ ] Domain層にインフラ依存（DB・外部API等）が混入していないか
- [ ] UseCase層がHTTPやgRPCの詳細に依存していないか
- [ ] 依存の方向が内側に向いているか（外→内）
- [ ] インターフェースを介さず具象に依存していないか

### ⚠️ エラーハンドリング
- [ ] `err` を無視していないか（`_` で捨てていないか）
- [ ] エラーに適切なコンテキストが付与されているか（`fmt.Errorf("...: %w", err)`）
- [ ] ドメインエラーとインフラエラーが適切に分離されているか
- [ ] エラーがそのままクライアントに露出していないか

### 📛 命名・可読性
- [ ] 変数名・関数名から意図が読み取れるか
- [ ] 省略しすぎていないか（`u` より `user` など）
- [ ] Goなら`MixedCaps`、proto/SQLはチームの命名規則に沿っているか
- [ ] コメントは「何をするか」でなく「なぜそうするか」を説明しているか
- [ ] 関数が単一責任になっているか（長すぎないか）

### 🧪 テスト
- [ ] ハッピーパス以外（エラー系・境界値）のケースがあるか
- [ ] テーブル駆動テストで書かれているか（Go）
- [ ] モックが適切に使われているか
- [ ] テスト名から何をテストしているかわかるか

### 🔒 セキュリティ
- [ ] SQLインジェクションの可能性はないか（プレースホルダを使っているか）
- [ ] 機密情報（トークン・パスワード等）がログに出力されていないか
- [ ] 認証・認可のチェックが漏れていないか
- [ ] 入力値のバリデーションがされているか

### ⚡ パフォーマンス
- [ ] N+1クエリが発生していないか
- [ ] ループ内で不要なDB/API呼び出しをしていないか
- [ ] 不要なメモリアロケーションがないか
- [ ] インデックスが適切に使われるクエリになっているか

---

## 言語・技術別 追加観点

### Go
- `context` が適切に引き回されているか
- goroutine のリークがないか（defer / cancel の忘れ）
- `interface` の定義は利用側に置かれているか
- スライスの初期化は `make` でサイズを指定しているか（パフォーマンス）
- YAGNI：今必要でない抽象化をしていないか

### gRPC / Protobuf
- フィールド名・型は命名規則に沿っているか
- フィールド番号は既存と衝突していないか
- `optional` / `repeated` の使い分けは適切か
- ステータスコードは適切か（`NotFound` / `InvalidArgument` など）

### SQL / DB
- トランザクション境界は適切か
- NULL許容カラムの扱いは適切か
- マイグレーションは後退可能か（ロールバックできるか）
- インデックスが効くWHERE句になっているか

### TypeScript / Vue / Nuxt / React / Next
- `any` を使っていないか（型を正しく定義しているか）
- コンポーネントの責務が適切に分割されているか
- 副作用（useEffect等）の依存配列は正しいか
- XSS・CSRFの考慮がされているか
- 不要な再レンダリングが発生していないか

```
## 🔴 Must（必ず修正）
- [ファイル名:行番号] 問題の説明 → 修正案

## 🟡 Should（できれば修正）
- [ファイル名:行番号] 問題の説明 → 修正案

## 🟢 Nice to have（提案）
- [ファイル名:行番号] 提案内容
```
